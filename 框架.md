> 核心理念：**人只负责"定义意图"和"验收结果"，中间一切交给 AI Agent + CI 自动闭环。**
>
> 效率思维：**每加一个流程步骤，先问自己——它是提高产出还是减少干预？两者都不是，就是无效开销。**
>
> 三个优化方向：
>
> - **多跑实验**：任务拆细、Agent 并行、睡觉时让 Agent 跑
> - **多自动化**：把你脑子里的验收标准变成断言，测试覆盖越多，你越能当甩手掌柜
> - **少干预**：更好的测试、更清晰的契约、CI 自动修复

## 第零部分：行为契约（任务启动协议）

### 为什么要先写契约

Agent 最贵的错误不是代码写错，而是**需求理解错了还写得很自信**。测试和实现都建立在错误的理解上，等发现时推倒重来成本极高。

行为契约是任务真正开始前的一次强制对齐——一份 5 分钟能写完、人类能快速确认的"我们约好做什么"。

**契约确认后：测试才能开始写。测试全绿后：实现才能开始写。**

### 什么时候写契约，什么时候不写

| 任务类型              | 要不要写契约 | 原因                            |
|-------------------|--------|-------------------------------|
| 新功能开发             | ✅ 写    | 意图最容易被 AI 误解的场景               |
| 现有功能改需求           | ✅ 写    | 需求变了，旧的"约定"已经失效               |
| Bug 修复            | ❌ 不写   | 走 Bug 修复流程 → `.claude/skills/bug-fix-flow.md` |
| 纯重构（行为不变）         | ❌ 不写   | 目标是"改完测试全绿"，测试本身就是契约          |
| 一次性脚本 / 临时工具      | ❌ 不写   | 生命周期太短，投入产出不对等                |

### 契约五要素

> 详细写作规范 → `.claude/skills/write-contract.md`

1. **前置条件** — 什么输入我才接受
2. **后置条件** — 我保证输出什么
3. **异常后置条件** — 失败时保证什么（错误类型、是否幂等、副作用是否已发出）
4. **不变式** — 任何情况下都必须成立的规则
5. **边界目录** — 边界情况怎么处理（含待确认项）

### 契约文件规范

> 模板 → `docs/contracts/_template.md`
> 自动创建 → `bash .claude/scripts/save-contract.sh <模块名> <功能名>`

- 命名格式：`YYYY-MM-DD_模块名_功能名.md`
- 状态流转：草稿 → 已确认 → 已实现 → 已废弃
- 存放路径：`docs/contracts/`

### 契约的生命周期

```
描述意图
   ↓
Agent 输出契约草稿（含 ⚠️ 待确认项）
   ↓
人类逐项确认，解决所有 ⚠️
   ↓
Agent 执行 save-contract.sh → 文件自动创建到 /docs/contracts/
   ↓
Agent 按契约写测试（先看到红）
   ↓
Agent 写实现（变绿）
   ↓
PR 创建 → link-contract.yml 自动关联契约 → qa-review.yml 自动触发 QA 审查
   ↓
PR 合并 → update-contract-status.yml 自动将状态改为「已实现」
```

**后续迭代时**：先改契约文件，走确认流程，再改测试，再改实现。契约永远是最高权威。

## 第一部分：AI 驱动的自动化测试体系

### 1. 思维转变

**认知一：测试写得越好，你就能给 Agent 越大的自主权，你就越能当甩手掌柜。**

**认知二：不要想"写测试"，要想"把你脑子里的验收标准翻译成代码"。** 你现在每天人工验收时脑子里在检查什么？把那些东西变成断言（assert），就是你的测试。

### 2. 代码架构要求

> 完整规范 → `CLAUDE.md` 代码架构约束章节

- **核心逻辑纯函数化**：逻辑和 I/O 分离，逻辑部分不依赖外部状态
- **I/O 层独立抽象**：数据库、网络、文件——全部抽成独立 service 层，通过接口注入
- **外部依赖 mock 化**：测试中使用 mock/stub，不依赖真实外部服务

### 3. 五层测试策略

> 测试写作规范 → `.claude/skills/write-tests.md`
> Bug 修复流程 → `.claude/skills/bug-fix-flow.md`

| 层级 | 策略 | 说明 |
|------|------|------|
| 第一层 | TDD by AI | 先写测试（红）→ 再写实现（绿），Prompt 已内嵌 CLAUDE.md |
| 第二层 | 回归测试 | 每个 bug 修复沉淀为永久回归测试，系统永远不会忘记 |
| 第三层 | AI QA 审查 | qa-review.yml 自动调用 AI 审查 PR diff，对照契约检查合规性 |
| 第四层 | E2E 测试（可选） | Playwright / Cypress 把人工验收路径代码化 |
| 第五层 | Mutation Testing（高阶） | Stryker (JS/TS) / mutmut (Python)，检验测试质量 |

**AI QA 审查说明：**

当前 `qa-review.yml` 通过 Anthropic API 调用单一 AI 模型审查。QA 结果作为 PR 评论发布，人类根据 severity 决定是否合并。

| QA 指出的问题类型           | 你的动作              |
|----------------------|-------------------|
| 真实缺陷（漏处理边界、潜在崩溃）    | 打回重改              |
| 值得改但不紧急             | 创建 Issue，当前 PR 可过 |
| 过度挑剔 / 不适用           | 忽略，直接合并           |

### 4. CI 流水线

> pre-commit hook → `hooks/pre-commit`
> CI 流水线 → `.github/workflows/test.yml`
> PR 关联契约 → `.github/workflows/link-contract.yml`
> 契约状态更新 → `.github/workflows/update-contract-status.yml`
> AI QA 审查 → `.github/workflows/qa-review.yml`

```
┌─────────────────────┐
│     Agent 提交代码    │
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│   pre-commit hook    │  ← 本地：提交前测试
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│   Lint + 类型检查     │  ← 第一道门
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│  单元测试 + 覆盖率门控  │  ← 第二道门（< 80% 自动拦截）
└──────────┬──────────┘
           ▼
┌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌┐
╎    E2E 测试（可选）     ╎  ← 第三道门（项目有 E2E 时启用）
└╌╌╌╌╌╌╌╌╌╌┬╌╌╌╌╌╌╌╌╌╌┘
           ▼
┌─────────────────────┐
│    构建 + 部署预览     │  ← 第四道门
└──────────┬──────────┘
           ▼
    ┌──────┴──────┐
    │             │
  ✅ 全部通过     ❌ 任一失败
    │             │
    ▼             ▼
 qa-review.yml  Agent 自动修复（最多 3 次，Prompt 级约束）
 触发 AI 审查     重新进入流水线
    │             │
    ▼         超过重试上限
 结果推送到        │
 PR 评论         ▼
    │         钉钉通知（WEBHOOK 未配置时跳过）
    ▼             │
  severity       ▼
  判断         等你介入
    │
    └──────→ 合并 → update-contract-status.yml 原子更新契约状态
```

**Agent 失败后的处理：**

| 失败类型                     | 你的动作                 |
|--------------------------|----------------------|
| 测试本身写错了（断言有误、mock 不对）   | 人工修正测试，重启 Agent      |
| 需求和实现有根本矛盾               | 回到契约，更新边界目录，重新确认     |
| 环境问题（依赖版本冲突、配置缺失）        | 修复环境配置，更新 CLAUDE.md  |
| Agent 能力边界（真的不会）          | 人工实现这部分，其余交回 Agent   |

### 5. 推荐的工具组合

| 场景               | 推荐工具                                   |
|------------------|----------------------------------------|
| 单元测试             | Vitest（JS/TS）、pytest（Python）           |
| E2E 测试           | Playwright（首选）、Cypress                |
| API 测试           | Supertest、httpx + pytest              |
| 快照测试             | Jest Snapshot、Playwright Screenshot   |
| QA 审查            | qa-review.yml（Anthropic API）           |
| Mutation Testing | Stryker（JS/TS）、mutmut（Python）          |
| CI 平台            | GitHub Actions                         |
| 代码质量             | ESLint、Biome、Ruff（Python）             |
| 覆盖率              | c8 / istanbul（JS）、coverage.py         |

## 第二部分：团队协作分支策略

### 分支规范

- **main** — 受保护分支，所有人（包括负责人和 Agent）通过 PR 合并，禁止直推
- **feature/功能名-操作人** — 开发分支，例：`feature/optimize-listing-ui-shaoquan`
- **fix/问题描述-操作人** — 修复分支，例：`fix/login-timeout-frank`

钉钉通知里 `github.actor` 自动显示操作人，出问题一眼定位。

**为什么不让所有人都推 main：** 两个 Agent 同时对 main 写入时，后一个会覆盖前一个的改动，没有任何警告。PR 流程强制让每次合并经过 CI 验证。

### 多人协调原则

**启动 Agent 前，在钉钉群里说一句**：「我在做 XX 功能，改 XX 模块」。低成本，防止任务重叠。

后续可以用 GitHub Issue 作为任务锁——每个任务对应一个 Issue，Agent 启动时自动 assign，别人看 Issue 列表就知道哪些模块在进行中。

## 第三部分：完整的开发闭环

> **关于多 Agent 并行**：当前推荐串行执行——等串行方案跑顺、瓶颈明确出现在等待时间时再引入 Agent Teams。

```
你（启动任务时）
  │
  ├── 1. 描述意图 → Agent 输出契约草稿
  ├── 2. 确认契约（解决所有 ⚠️）→ Agent 自动存档到 /docs/contracts/
  ├── 3. 启动 Claude Code（串行执行）
  └── 4. 去做别的事 / 睡觉

Claude Code 自动串行执行
  │
  ├── 自主分解任务
  ├── TDD：先写测试（红）→ 写实现（绿）
  ├── CI 失败 → 读日志 → 修复 → 重试（最多 3 次，Prompt 级约束）
  ├── 超出重试上限 → 钉钉通知你介入
  └── 完成后创建 PR

PR 创建后自动触发
  │
  ├── link-contract.yml → 自动关联契约文件
  ├── CI 跑测试门控
  │     └── 失败超过重试上限 → 钉钉通知
  └── qa-review.yml → AI 审查 diff + 契约
        └── 结果推送到 PR 评论

你（回来验收时）
  │
  ├── 看钉钉：有没有失败通知（没有 = 一切正常）
  ├── 看 PR 评论：QA severity 判断要不要改
  ├── 花 10 分钟 review 关键决策
  └── 合并 → update-contract-status.yml 自动更新契约状态

多人场景补充
  │
  ├── 启动前在钉钉群说一句「我在做 XX，改 XX 模块」
  ├── 在自己的 feature/功能名-姓名 分支上工作
  └── 负责人看各人 PR + QA 审查意见，决定合并
```

## 第四部分：Agent 的数据化管理

### 1. CI 指标收集

> 指标收集 → `test.yml`（上传为 Artifact，保留 90 天）
> 下载汇总 → `bash .claude/scripts/download-metrics.sh`
> 统计报告 → `bash .claude/scripts/report.sh [天数]`

**注意**：GitHub Artifact 保留 90 天后自动删除。建议定期运行 download-metrics.sh 汇总到本地。

### 2. Agent 绩效指标

| 指标          | 含义               | 优化方向              |
|-------------|------------------|-------------------|
| Token 消耗量   | 这个任务花了多少钱         | 同等质量下越少越好         |
| 任务完成率       | 分配 10 个任务完成了几个    | 越高越好              |
| 首次通过率       | 提交后 CI 一次通过的比例    | 反映代码质量            |
| 自修复成功率      | CI 失败后自己修好的比例     | 越高 = 越少需要你介入      |
| 平均完成时间      | 从接受任务到 PR 就绪      | 用于估算效率            |
| Bug 引入率     | 新代码引入了多少新 bug     | 越低越好              |
| 测试覆盖增量      | 这次提交增加了多少覆盖率      | 应持续增长             |
| QA 问题发现率    | 每次审查平均发现几个问题      | 持续下降说明代码质量在提升     |

### 3. 用数据优化 Agent 配置

有了数据后，可以做实验：

- **Prompt A vs Prompt B**：同样的任务，哪个 prompt 让 Agent 首次通过率更高？
- **模型对比**：Opus vs Sonnet 在你的项目上，性价比哪个更高？
- **QA 模型效果**：不同模型在你们项目上，哪个发现的问题更有价值？

## 附录：关键心智模型速查

| 心智模型       | 一句话解释                                        |
|------------|----------------------------------------------|
| 契约先行       | 需求对齐发生在写代码之前；bug 修复和纯重构不需要契约                  |
| 五要素契约      | 前置/后置/异常后置/不变式/边界目录，失败行为和成功行为同等重要           |
| 自动化存档      | 契约确认后由 save-contract.sh 自动创建文件，无需手动操作         |
| 验收标准代码化    | 你脑子里验收时在检查什么，就把什么变成断言                        |
| 覆盖率硬门控     | CI 配置 --cov-fail-under=80，低于阈值自动拦截，不依赖人记得检查   |
| Bug = 组织记忆 | 每个 bug 修复沉淀为一条回归测试，系统永远不会忘记                   |
| AI QA 审查    | qa-review.yml 自动调用 AI 审查 PR diff，结果发布为 PR 评论    |
| 只通知失败      | 群里没消息 = 一切正常；有消息 = 需要处理                       |
| 契约状态自动同步   | PR 合并后通过 API 原子更新契约状态                          |
| 指标驱动优化     | CI 指标上传为 Artifact，download-metrics.sh 汇总，report.sh 统计 |
