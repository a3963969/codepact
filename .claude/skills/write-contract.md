# 如何写符合规范的行为契约

## 契约五要素

**前置条件**：描述什么输入才合法。
- 数据类型约束（字符串、数字、对象结构）
- 范围约束（长度、大小、格式）
- 系统状态约束（数据库连接、依赖服务可用）

**后置条件**：成功时保证输出什么。
- 返回值的结构和类型
- 系统状态的变化（数据库写入、缓存更新）
- 副作用（邮件发送、日志记录）

**异常后置条件**：失败时保证什么。
- 返回的错误类型及错误码范围（如 400/404/500）
- 事务是否已回滚，有无部分写入
- 外部副作用是否已发出（邮件是否已发、消息队列是否已推送）
- 幂等性：同一请求重试是否安全

**不变式**：任何情况下都不能违反的规则。
- 数据一致性约束
- 安全约束（密码不落库、敏感数据加密）
- 业务规则（同一邮箱不能注册两次）

**边界目录**：列出所有边界情况和处理方式。
- 已知边界：明确处理方式
- 未知边界：标注 ⚠️ 待确认，等人类决策

## 质量标准

✅ 好的契约：
- 5 分钟内能读懂
- 边界目录覆盖所有能想到的异常场景
- 待确认项明确标注，不含糊

❌ 坏的契约：
- 描述"如何实现"而不是"做什么"
- 边界目录为空
- 使用模糊词汇（"适当地"、"合理地"）
