# 项目规范

## 初次使用

克隆仓库后，运行初始化脚本安装 pre-commit hook 和创建必要目录：

```bash
bash .claude/scripts/setup.sh
```

## 任务启动协议（必须遵守）

接到任何新功能任务，在写任何测试或代码之前，先输出行为契约：

在写任何测试或代码之前，先输出一份行为契约，包含：
- 前置条件：什么输入才合法
- 后置条件：成功时保证返回什么
- 异常后置条件：失败时保证什么（错误类型、是否幂等、副作用是否已发出）
- 不变式：任何情况下都不能违反的规则
- 边界目录：列出你打算如何处理边界情况，不确定的标注「⚠️ 待确认」

等人类确认契约后，执行以下命令保存契约文件：
bash .claude/scripts/save-contract.sh <模块名> <功能名>

然后再开始写测试。

以下任务不需要写契约，直接走对应流程：
- Bug 修复 → 走「Bug 修复流程」
- 纯重构（行为不变）→ 确保测试全绿即可
- 一次性脚本 → 直接实现

## 代码架构约束

- 所有核心逻辑必须是纯函数，不依赖外部状态
- 所有 I/O 操作必须抽象为独立 service 层
- 所有外部依赖通过接口注入，测试中使用 mock

## 测试规范

- 所有新功能必须先写测试，再写实现（TDD）
- 测试覆盖范围：
  - 正常路径（happy path）
  - 边界条件（空值、极大值、极小值）
  - 错误处理（网络超时、权限不足、数据格式错误）
  - 契约中的所有边界目录条目
- 提交前必须运行测试，全部通过才能提交
- 测试覆盖率目标：语句覆盖 > 80%（由 CI 门控强制执行）
- 核心业务逻辑（计算、状态变更）要求 > 95%
- 遗留项目：以"当前覆盖率 + 5%"为阶段目标，禁止新代码让覆盖率下降

## Bug 修复流程

1. 先写一个能复现 bug 的测试用例（确认测试失败）
2. 修复代码
3. 确认测试通过
4. 这条回归测试永久保留，不可删除

## CI 失败处理

1. 阅读完整的错误日志
2. 定位失败的测试或构建步骤
3. 修复问题并重新提交
4. **本条是行为指引，不是 CI 自动化**：Agent 操作者在收到 CI 失败通知后，应主动阅读日志、修复问题并重新提交，最多尝试 3 次（复杂任务可调整为 5 次，简单修复若 1 次未过直接输出报告）。CI 本身不会自动触发 Agent 重试。
5. 重试上限后仍失败，输出详细失败分析报告等待人类介入

## 分支规范

- 所有代码变更通过 PR 合并到 main，禁止直接推送 main（包括负责人和 Agent）
- 在 feature/功能名-姓名 分支上工作，通过 PR 合并
- 启动任务前在钉钉群说明「我在改 XX 模块」，避免任务重叠
